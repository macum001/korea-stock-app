# 공시 AI 분석 화면 재설계 설계서

> 작성: 한국 주식 공시 앱 (korea-stock-app)
> 목적: 모든 공시 종류를 **하나의 틀(슬롯 구조)**로 담고, 투자자가 **놓치는 데이터**를 노출한다.
> 대상 파일: `src/components/disclosure/DisclosureSummarySheet.tsx` (674줄, 여러 화면 공유)

---

## 핵심 원칙

1. **공시 종류마다 화면을 따로 만들지 않는다.** 9종류 × 각각 디자인 = 관리 지옥.
2. 모든 공시는 **같은 6개 슬롯** 뼈대를 쓴다.
3. **3초 / 30초 / 3분 계층**:
   - 3초 — 판정(호재/악재/주의 + 한 문장)
   - 30초 — 핵심 숫자 + 가장 중요한 신호
   - 3분 — 펼쳐서 상세 (탭 / 그리드)
4. 정보를 버리지 않되, 중요도에 따라 **위계**를 둔다 (덜 중요한 건 접거나 그리드로 압축).

---

## 슬롯 구조 (모든 공시 공통 뼈대)

| 슬롯 | 내용 | 성격 | 데이터 출처 |
|---|---|---|---|
| ① 판정 + 한 문장 | 호재/악재/주의 배지 + 결론 한 줄 | **고정** | AI `impactLabel`, `summary` |
| ② 핵심 숫자 2~3개 | 종류별 KPI 카드 | **종류별** | AI `keyNumbers` |
| ③ 종류별 시각화 | 막대/도넛/게이지 등 (없으면 생략) | **종류별** | `keyNumbers` 가공 |
| ④ 꼭 알아야 할 신호 | 재무 경고(빨강) / 안심(초록) | **고정** | 재무 신호 데이터 |
| ⑤ AI 상세 (탭) | 요약 · 투자포인트 · 리스크 · 일정 | **고정** | AI `detail`,`risks`,`timeline` |
| ⑥ 회사 기본정보 (2×2 그리드) | 최대주주 · 자본금 · 배당 · 출자 | **고정** | DART 정형 데이터 |

**①④⑤⑥ = 고정** (모든 공시 동일)
**②③ = 종류별로 갈아끼움**
→ 새 공시 종류가 와도 ②③만 정의하면 화면 완성.

---

## 종류별 ②③ + 놓치면 안 되는 데이터

| 공시 종류 | ② 핵심 숫자 (KPI) | ③ 시각화 | ⚠️ 놓치면 안 되는 것 |
|---|---|---|---|
| 분기·실적 | 매출, 영업이익 + 평균比 | 사업부문 막대 | 적자전환, 특정부문 손실 |
| 유상증자 | 발행가, 조달액, 희석률 | 희석 도넛(기존:신주) | **자금용도**(채무상환=경고), 제3자배정 대상 |
| 전환사채(CB/BW) | 발행액, 전환가 | 전환 시 희석 막대 | **리픽싱 조건**(주가하락시 추가희석), 몇 번째 발행 |
| 지분변동 | 변동 수량, 보유비율 | 전→후 비율 바 | **보유목적**(경영참가=경영권 신호) |
| 자사주 | 취득 규모, 방법 | — | **소각 여부**(소각=호재 / 처분=악재) |
| 배당 | 주당배당금, 시가배당률 | 배당률 게이지 | 배당성향, 전년比 증감 |
| 합병·분할 | 합병비율 | — | **반대주주 매수청구가** |
| 계약·수주 | 계약금액 | 연매출 대비 % 바 | 계약상대, 기간 |
| 소송 | 청구금액 | 자기자본 대비 % | 소송 상대, 패소 시 영향 |

**"놓치면 안 되는 것"이 이 프로젝트의 핵심 가치** — 원문엔 있지만 그냥 보면 안 와닿는 것:
- 증자 **자금용도**: 시설투자=호재 / 채무상환=위험
- CB **리픽싱**: 주가 하락 시 추가 희석되는 독소조항
- 지분변동 **보유목적**: "경영참가"면 경영권 분쟁 신호
- 자사주 **소각 vs 처분**: 정반대 의미

## 상세 영역 구성 (⑤⑥) — 아코디언 대신

- **⑤ AI 해석** = 탭 (요약 / 투자포인트 / 리스크 / 일정)
- **⑥ 회사 기본정보** = 2×2 그리드 (최대주주 / 자본금 / 배당 / 출자)
- AI 해석(읽는 것)과 회사 기본정보(참고 숫자)를 **성격별로 분리**.
- 회사 기본정보는 펼치는 게 아니라 작은 숫자 카드로 바로 노출.

## 백엔드 현황 (중요)

진짜 분석 진입점: `analyzeByReceiptNo` (`backend/src/services/ai/receiptAnalysis.service.ts`)
→ **이미 원문 기반 + 유형별 체크리스트 + keyNumbers 구조화 완성형.**

- 원문 추출: `extractDisclosureCore` (`dartDocument.service.ts`) — 작동 중
- 출력 필드: `summary, detail, reason, impact, impactLabel, risks[], keyNumbers[], timeline, subtype`
- 유형별 체크리스트 `getSubtypeChecklist` — **8종 보유**:
  전환사채/유상증자/무상증자/감자/자사주/실적/계약·수주/합병·분할

### 백엔드 빈틈 (체크리스트 없음 → 추가 필요)
- **지분변동** (DB 최다, 239,219건!) ← 우선순위 1
- **배당** (184,158건)
- **소송**

→ 프론트는 데이터를 거의 다 받고 있음. **화면 로직만 바꾸면 됨.**
→ 백엔드는 위 3종 체크리스트만 추가하면 됨.

## 구현 순서 (안전하게 단계별)

각 단계: 적용 → 실제 공시로 검증 → 커밋. 하나 잘 되면 다음.

- [ ] **0단계** — 이 설계서 커밋 (지도 박제) ← 지금
- [ ] **1단계** — 공통 틀: ① 판정 배지 + 결론 / ④ 신호 / ⑤ 탭 / ⑥ 그리드 (종류 무관)
- [ ] **2단계** — 실적 종류 ②③ (가장 흔함, 시각화 효과 큼: 사업부문 막대)
- [ ] **3단계** — 증자/지분변동 ②③ + 백엔드 지분변동 체크리스트
- [ ] **4단계** — 나머지 종류 ②③ (배당/CB/계약/소송 등) + 백엔드 배당·소송 체크리스트
- [ ] **5단계** — 다듬기 (색/문구/판정 로직)

### 절대 원칙
- 한 번에 674줄 다 바꾸지 않는다.
- 잘 돌아가는 기존 분석을 망가뜨리지 않는다 (각 단계마다 검증).
- 데이터는 이미 백엔드에 있다 → 프론트 화면 로직 변경이 주 작업.

## [4단계 진단 결과 — 데이터 추출 빈틈 확정] (작성: 진단 완료)

### 문제
배당·증자 공시에서 핵심 숫자가 keyNumbers에 "원문에 없음"으로 누락. **실제 DART 원문엔 명확히 존재함.**

### 확정 증거
- 배당(메타바이오메드 20260625900295): 원문에 `1주당 배당금 55원`, `시가배당률 1.5%` 있으나 앱은 "원문에 없음"
- 증자(오에스피 20260626000347): 원문에 `신주 발행가액 1,632원`, `할인율 -10%`, `증자전 발행주식총수 7,283,547주` 있으나 앱은 발행가/할인율 "원문에 없음"

### 원인 (추정 → 확정 필요)
`backend/src/services/ai/dartDocument.service.ts`의 `extractCoreSection`이 핵심 숫자를 못 뽑음.
라벨(indexOf 부분매칭)은 "발행가액"⊂"신주 발행가액"이라 잡혀야 정상인데 안 잡힘 →
- 가설1: reportName 매칭 실패로 fallback行
- 가설2: fetchDartDocumentText 원문 추출 실패
- 가설3: 원문은 갔으나 AI가 표를 keyNumbers로 못 만듦

### 다음 작업 (우선순위 높음)
1. receiptNo로 extractDisclosureCore 실제 호출해 mode(section/fallback/none)와 추출 텍스트 확인 → 가설 확정
2. 원인별 수정 (라벨 보강 / 원문추출 디버그 / 프롬프트 강화)
3. 재분석으로 검증 (발행가·시가배당률 나오는지)
4. 데이터 확보 후 → 증자 희석률 시각화 (신주÷(기존+신주), 오에스피=16.7%), 배당 시가배당률 게이지

### 시각화는 데이터 수정 후에 (교훈)
화면 잘 만들어도 데이터 비면 빈 깡통. 추출 개선이 시각화보다 선행되어야 함.
